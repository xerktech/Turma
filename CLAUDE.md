# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Repo Is

AgentHub is the source and CI for the Claude Code agent fleet used with the TrueNAS-based home lab: a **one-container-per-host** agent image that scans a git root and multiplexes many worktree-backed Claude Code Remote Control sessions, and a central dashboard ("agent-hub") that lists each host's repos, spawns/kills those sessions, and monitors them. It builds two images and pushes them to GHCR; the running stack is deployed from the sibling **DockerOps** repo (`compose/agents-truenas.yaml`, deployed via Portainer GitOps).

**Session model (post-redesign).** There is one agent container per physical host, mounted at a git root (`REPOS_ROOT`, e.g. `/mnt/data/Docker/git`). The container scans the root one level deep for git repos and reports them; from the hub UI you pick a repo and spawn a **session**, which the container backs with a git worktree (branch `agent/<id>`, or a custom name the composer still forces under the `agent/` prefix, optionally forked off a chosen base branch) under `REPOS_ROOT/.agenthub/worktrees`, running its own `claude --remote-control` in its own tmux + loopback ttyd (with an optional initial task prompt delivered as claude's positional prompt, and an optional `--model`/`--permission-mode` from the composer). Many sessions run concurrently (up to `MAX_SESSIONS`), including several on one repo via separate worktrees, and each registers in claude.ai/code as `<host>-<repo>-<worktree-or-label>`. All spawn options are validated agent-side (allowlisted branch/base refs, fixed model/permission enums) so nothing free-form reaches the shell; the `agent/<id>` branch and `agent-<id>` tmux stay the canonical internal keys and a label is presentational only. Sessions are spawned/killed/started/restarted/deleted from the hub; **killing a session removes it from the hub (worktree deleted, registry record dropped) but keeps its git branch, conversation, and token-usage history** (transcripts live under `~/.claude/projects`, keyed by worktree path, separate from the worktree files) — each repo's "Resume" picker lists its 5 most recently killed sessions (`~/.agenthub/closed.json` on the agent) and can bring one back on the same branch with its conversation resumed via `claude --resume <newest transcript id>`; `delete` goes further and also removes the branch — `git branch -D` when fully pushed/merged, otherwise parked as `agent-trash/<id>-<epoch>` for a `TRASH_RETENTION_DAYS` grace period (default 7) before being pruned — which forfeits resume. This replaced the old model of one fixed-repo container per session.

## Repository Structure

- `agent/` — Per-host headless agent image (currently Claude Code; the name is agent-generic so it can host other agents later). `hub-agent.py` is the **session manager and heartbeat in one process**: it scans `REPOS_ROOT` for repos, owns a persisted session registry (`~/.agenthub/sessions.json`), executes hub-issued commands that ride back on the heartbeat reply (spawn/kill/start/restart/resume/delete, at-least-once with `cmdId` de-dup), drives each session's worktree + tmux + ttyd, and heartbeats the repo list and one record per session (git state, per-session token usage, live-session signals — transcript freshness, pending questions, new PR links) plus a container-log tail. On boot it auto-resumes sessions that were `running`. Also `entrypoint.sh` (creds preflight, then launches the tunnel and `exec`s the session manager as PID 1 — the container stays up with zero sessions) and `tunnel-agent.js` (the reverse tunnel; the hub's `{open,port}` selects which per-session ttyd to bridge, over one per-host control channel).
- `agent-hub/` — Central dashboard for the per-host agent containers (https://agents.xerktech.com via the Cloudflare tunnel; port 8300 on the LAN): a **host → repo → session tree** — each host expands into its scanned repos (each with a one-click "+ New session" button — instant bare spawn with today's defaults — plus a ▾ caret opening a progressive-disclosure "New session" composer for an optional initial task prompt, a human-friendly label, and spawn options (base branch, custom branch name, model, permission mode, last-used options remembered per repo in `localStorage`); and, when it has recently killed sessions, a "Resume" picker of the last 5), each repo into its worktree sessions with working/idle/waiting-on-question state, branch, per-session token usage parsed from that worktree's `~/.claude/projects` transcripts, and per-session Attach / Restart (clear context) / Kill / Start / Delete. Session commands are queued on the hub and drained via the heartbeat reply. Pushes edge-triggered alerts to the self-hosted ntfy on topic `agents` (host offline/recovered, restart loop, daily cost threshold summed across a host's sessions, per-session turn finished / question waiting / PR created). UI, API, and the click-to-attach live terminal (`/term/<sessionId>/`, reverse-tunneled to that session's ttyd by port) sit behind single-user HTTP Basic auth (`HUB_USER`/`HUB_PASSWORD` on the hub service); agents authenticate heartbeats, tunnel WebSockets, and ttyd with one shared token (`HUB_TOKEN` in the agent's env = `HUB_AGENT_TOKEN` on the hub) — all set inline in DockerOps' `compose/agents-truenas.yaml`.
- `.github/workflows/` — GHCR image builds (see Build & Deploy below).

## Build & Deploy

- `.github/workflows/agent-image.yml` builds `ghcr.io/xerktech/agent` on any change under `agent/**`. The primary version tag IS the bundled Claude Code release (pinned into the build via the `CLAUDE_CODE_VERSION` build-arg so tag and contents can't drift); `github.run_number` disambiguates repo-side rebuilds of the same version.
- `.github/workflows/agent-hub-image.yml` builds `ghcr.io/xerktech/dockerops-agent-hub` on any change under `agent-hub/**`. No upstream package to version against, so the build counter is the version.
- Both push `:latest`, a versioned/build tag, and `:sha-<sha>`; Watchtower keeps `:latest` current on the host. The DockerOps `compose/agents-truenas.yaml` references `ghcr.io/xerktech/agent:latest` — keep that image ref in sync if you ever rename it here.
- **Deployment lives in the DockerOps repo**, not here: `compose/agents-truenas.yaml` defines the `agent-hub` service and a single per-host `agent-host` container (mounted at `REPOS_ROOT`, `MAX_SESSIONS`/`TTYD_PORT_BASE`, host mounts, the shared `HUB_TOKEN`/`HUB_AGENT_TOKEN`, ntfy publisher creds, basic-auth, optional `PRICING_JSON` — inline-JSON per-MTok rates consulted only for models the built-in pricing table doesn't match; unmatched models are reported as `usage.unpricedModels` and flagged ⚠ in the UI). Because one container now hosts many concurrent Claude sessions, its `mem_limit`/`cpus`/`pids_limit` are sized against `MAX_SESSIONS` rather than a single session. Editing image content here + pushing rebuilds the image; changing how it's run (or adding another host) means editing that compose file in DockerOps.
- **PR gates (pre-merge to main).** The build workflows above run only post-merge; these run on `pull_request` → `main` (open + each push) and block the merge on findings: `code-scan.yml` (Semgrep SAST over the JS/Python + Dockerfiles + secret patterns, hadolint on both Dockerfiles, ShellCheck on `entrypoint.sh`); `agent-image-scan.yml` / `agent-hub-image-scan.yml` build each image locally (no push) and Trivy-scan it for CVEs + secrets (`ignore-unfixed`, HIGH/CRITICAL gate), path-filtered to their folder like the build workflows. Because the images bundle third-party binaries (docker CLI, `gh`, ttyd, npm), keep the pinned tool versions current — that's how most CVEs are cleared; genuinely non-actionable upstream base-image findings go in the root `.trivyignore` (a reviewed triage list, each with a reason), and anything unlisted still fails the scan.
- **All workflows run on the home-lab self-hosted runners** (`runs-on: [self-hosted, linux]`). Constraints that shape the jobs (established in the sibling SwitchBoard repo): every job starts with a "Reset workspace ownership" step (Docker steps leave root-owned files on the persistent runner); no passwordless sudo, so CLI tools (hadolint, Trivy) are installed to `$HOME/.local/bin` + `$GITHUB_PATH` rather than via their Docker/ghcr actions; the runner can't authenticate to ghcr.io for pulls (Docker Hub pulls and authenticated ghcr **pushes** work); there's no GitHub Advanced Security, so there is no code-scanning API — findings live in the job log and `--exit-code` is the gate (no SARIF upload). `npm` isn't on the runner, so the Claude Code version lookup runs in a throwaway `node:24-alpine` container.

## Conventions

- All credentials are inline in environment variables (no Docker secrets mechanism) — this matches the DockerOps convention. The live secrets (`HUB_TOKEN`, `HUB_AGENT_TOKEN`, basic-auth, ntfy) are set in DockerOps' `compose/agents-truenas.yaml`, not in this repo.
- Each session runs as root (needs full host access) as an interactive `claude --remote-control` with `--permission-mode bypassPermissions`, which Claude Code refuses under root unless `IS_SANDBOX` is set — set in the compose env. (It's deliberately the interactive form, not `claude remote-control` server mode: server mode's terminal is a QR/status lobby with no conversation, so the Agent Hub's live terminal would have nothing to show or type into.) Sessions are now independent processes inside the one host container, so a session ending no longer restarts the container — the manager just marks it stopped; "Restart (clear context)" relaunches a single session's Claude in place.
- All of a host's sessions share the one mounted `~/.claude` login; distinct worktree paths give each its own project slug and Remote Control bridge pointer, so concurrent sessions don't collide. `MAX_SESSIONS` caps concurrency (shared-login contention + resources), and the manager staggers session launches slightly on boot.
- Agents connect purely outbound to the public `HUB_URL` (the Cloudflare tunnel), so they work from any host/network, not just the hub's.
